Current: JS-level scoping (no RLS yet)
Every query manually scoped in hooks. RLS adds DB-level enforcement in Phase 3.
// src/hooks/useEvents.js — ALL queries follow this pattern async function fetch() { const { data } = await supabase .from('events') .select('*') .eq('boutique_id', boutique.id) // ALWAYS scoped — never omit .order('date', { ascending: true }) if (data) setEvents(data) } // boutiqueId always from useAuth() — never from user input async function createEvent(payload) { return supabase.from('events').insert({ ...payload, boutique_id: boutique.id // injected from auth context }).select().single() }
Phase 3 target: Row Level Security
Applied to every table. Uses boutique_members, not Clerk org_id.
-- Helper function (created once) CREATE OR REPLACE FUNCTION is_boutique_member(target_boutique_id uuid) RETURNS boolean LANGUAGE sql STABLE SECURITY DEFINER AS $$ SELECT EXISTS ( SELECT 1 FROM boutique_members WHERE boutique_id = target_boutique_id AND user_id = auth.uid() -- Supabase Auth, not Clerk ); $$; -- Applied to every table ALTER TABLE events ENABLE ROW LEVEL SECURITY; CREATE POLICY "boutique_isolation" ON events FOR ALL USING (is_boutique_member(boutique_id));

Authentication & authorization
Supabase Auth JWT on every request — anon key never exposes service role
boutique.id always resolved from boutique_members via auth.uid() — never from user input
Role model: owner / coordinator / front_desk / seamstress / decorator (stored in boutique_members)
Role enforcement currently in UI only — DB-level role checks coming in Phase 3
No RLS yet — JS scoping is the only isolation layer until Phase 3
Module manager restricted to owner role — enforced in UI today, DB in Phase 6
Data security
All queries scoped by boutique_id in hooks — no cross-tenant data possible via app code
No RLS — DB-level isolation is a gap until Phase 3
Supabase anon key is safe to expose — it can only access data RLS allows
Service role key never used in frontend — only in server-side scripts/webhooks
Environment variables in Vercel encrypted secrets — not in codebase
No PII scrubbing in error logs yet — Sentry/error monitoring not set up
Input validation
Client-side JS validation in hook functions before DB writes
No Zod schemas yet — validation is ad-hoc JS checks. Zod planned for Phase 3.
SQL injection impossible — supabase-js uses parameterized queries only
React auto-escapes all rendered values — no XSS from data
Webhook signature verification not yet implemented (no webhooks yet)
No rate limiting on Supabase queries from client yet
